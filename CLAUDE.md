# ルール
## 操作範囲
操作対象は必ずこのフォルダ(カレントディレクトリ)内のみ
上位フォルダには絶対に触らない
デスクトップやドキュメントフォルダは操作禁止
## 全般的な振る舞い
- 常に日本語で応答してください。
- 修正を行う前に「何をどう変えるか」の計画を提示し、承認を得てから実行してください。
- 一度に修正するファイルは最大3つまでとし、段階的に進めてください。


## 品質管理
- コードを生成する際は、エラーハンドリングを省略せずに記述してください。
- 未使用のインポートや変数が残らないように清掃してください。
- Windows環境特有のファイルロックエラー（OneDrive等）に配慮してください。


## コミュニケーション
- 専門用語を多用しすぎず、分かりやすい解説を心がけてください。
- 実装が完了したら、動作確認のために実行すべきコマンドを具体的に提示してください。
- 新しい概念や複雑な文法（SignalsやRiverpodのGeneratorなど）を使う場合は、何をしているのかコメントで補足してください。

## レビュー
- こちらが「コメント付きで」と指示したらレビューに加え、コードごとにコメントの解説を付けてください。
- また、こちらが自主的にコメントを書いていた場合、正誤を判断し、正しい場合はそのまま、間違っている場合は指摘、変更してください。

## セッション運用
- セッションが一区切りついたら会話履歴を /clear でクリアする運用にしています。
- そのため、進捗・設計判断・つまづきポイント・未対応の穴埋め ToDo は memory システム (MEMORY.md とその参照先) に必ず残してから区切ってください。
- 再開時はまず関連 memory を読み、続いて現在のコード状態を確認してから作業に入ってください。
